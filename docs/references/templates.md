# Templates

路由质量工程中使用的标准模板。

---

## 1. 证据卡片

用于 Phase 1 收集证据时，对每个问题 case 的标准化记录。

```
Case: [来源标识，如 截图-13:07 / test#14 / 用户反馈]
素材: [几张什么图/视频，semantic_label]
文本: [用户原文]
预期路由: [应该走哪个 skill]
实际路由: [实际走了哪个 skill]
Plan log: [planner 输出的 user_intent 和 skill_name]
```

---

## 2. 修改 Spec

用于 Phase 3，在动手改代码之前记录方案供审核。

```markdown
# {标题}

## 问题总结
[证据卡片 + 根因分类，来自收集和定位阶段]

## 修改方案

### 修改 N: {文件名} — {改什么}
**改前**：[原文引用]
**改后**：[修改后的文字]
**根因**：[对应 A-F 哪个类型]

## 不改的部分
[明确列出不改什么、为什么不改]

## 测试计划

### 新增测试用例
| case_id | 素材 | 文本 | 预期路由 | 测试目的 |
|---------|------|------|---------|---------|

### 验证流程
1. 跑基线（改前）
2. 应用改动
3. 跑验证（改后）
4. 对比 before/after
5. 准确率目标：>= {当前准确率}，且失败 case 数 <= {当前失败数}
```

## 3. 版本快照目录结构

每层完成后保存到 `.routing-quality/snapshots/v{N}-{layer-name}/`：

```
v{N}-{layer-name}/
  skills/                    # 所有 skill description 的完整副本
    {skill-1}.md
    {skill-2}.md
    ...
  test-cases.json            # 本层测试用例
  test-results.json          # 测试结果
  failed-cases.json          # 失败 case 列表（传递给下一层）
  summary.md                 # 本层改动摘要
```

### summary.md 结构

```markdown
# Layer {N} Summary

## 改动内容
[改了哪些 description，具体修改了什么]

## 改动原因
[对应的根因分类和分析]

## 测试结果
- 改前准确率: X%
- 改后准确率: Y%
- 改善的 case: [列表]
- 新增失败: [列表，应为空]

## 传递给下一层的失败 case
[failed-cases.json 的可读版本]
```

## 4. 变更记录

用于被动修复完成后，记录本次改动的完整上下文。

```markdown
## 变更概览
[一段话总结：改了什么、为什么、效果]

## 修改的文件和内容
[每个文件的具体改动摘要]

## 测试结果

| 指标 | 改前 | 改后 | 判定 |
|------|------|------|------|
| 总准确率 | X% | Y% | Y >= X |
| 完全失败 case 数 | N | M | M <= N |
| 不稳定 case 数 | P | Q | Q <= P |
| 新增 case 通过率 | - | Z% | Z = 100% |

## 剩余失败 case 分析

| Case | 素材 | 用户文本 | 预期 | 实际偏差 | 频率 | 原因分析 |
|------|------|---------|------|---------|------|---------|
```

## 6. Baseline Intent Spec

用于 Phase 0 收集时，从用户/产品视角定义每个 skill 的路由预期。独立于 description，作为后续所有层的 test case 来源。

### 每个 Skill 的意图定义

| 字段 | 内容 |
|------|------|
| Skill 名称 | {name} |
| 一句话意图 | 用户想做什么时应该走这个 skill？ |
| 必须通过的场景 | 3-5 个最典型的输入（素材+文本+预期路由） |
| 绝对不该走这里的场景 | 2-3 个容易误路由到这里的输入 |
| 已知模糊区域 | 哪些输入你自己也拿不准？ |
| 竞争 skill | 跟谁最容易混？混了的话哪个错误更可接受？ |

---

## 5. 达标门槛

测试结果的通过标准：

- 总准确率不低于改动前
- 目标失败 case 改善（至少从 0/3 → 2/3 或 3/3）
- 无新增完全失败 case
- 不稳定 case（3 次中仅 1 次失败）：记录但不阻塞
